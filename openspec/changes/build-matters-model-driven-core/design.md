## Context

Matters v0.2 is a local-first application for turning the user's authorized
content universe into traceable sources, evidence, candidate people and events,
Matters, open loops, outcomes, and bilingual views. The content universe
includes local user-content roots, documents, PDFs, spreadsheets, photos and
images, Gmail messages/threads/attachments, reachable cloud-backed user
content, and explicit pasted input. "All sources" means that every enumerated
item receives a declared terminal disposition; it does not mean blindly
executing files, loading credentials, following arbitrary links, or claiming
success for unavailable cloud placeholders.

The current repository already contains M0 plus C1-C12, model-derived
contracts, a Python in-memory synthetic core, a dormant generic Jira adapter,
and saved G0-G8 evidence. A read-only audit found that those plans and receipts
are scoped to selected text/Jira-shaped synthetic cases and are stale for the
new source universe. The product surface is not usable, AI/local-skill execution
is not implemented, correction requests do not dispatch original-owner
recomputation, durable source discovery is absent, public evidence contains
machine-local paths, required blob source is ignored, and G8 fingerprints
include generated bytecode.

Four authorities remain separate:

1. OpenSpec owns product scope, requirements, and acceptance.
2. `BehaviorCommitmentLedger` owns plane-partitioned external promises and maps
   each promise to one primary model and one primary path.
3. FlowGuard owns finite state, retry, rejection, failure, correction,
   parent/child closure, field lifecycle, UI behavior, and evidence freshness
   models.
4. CodeContracts, Model-Test Alignment, and TestMesh own implementation and
   test evidence.

The v0.2 main path requires no Jira or Rovo access. Jira remains a dormant,
read-only, post-v0.2 provider adapter and cannot contribute to v0.2 acceptance.
The user has authorized all reachable user-owned content for the first private
run. That standing source grant permits staged read and analysis, but it does
not authorize Gmail mutations, external messages, file deletion, execution, or
public transfer. Raw and derived private material remains outside the
repository.

## Goals / Non-Goals

**Goals:**

- Reuse M0 and C1-C12 as the only canonical behavior owners.
- Authorize a finite source universe made from declared user-content roots,
  Gmail read scope, pasted input, and configured cloud-backed content roots.
- Freeze a candidate-scope manifest and versioned tracking policy before broad
  analysis; separate `hard_excluded` system material, generated clutter, junk,
  irrelevant items, unavailable inputs, and `tracked` items without losing
  disposition history or waiting for ordinary user classification.
- Discover and disposition local text/code/config, PDF/Office/spreadsheet,
  image/photo, Gmail/thread/attachment, audio/video/archive metadata, cloud
  placeholders, unsupported files, and inaccessible items in resumable stages.
- Distinguish metadata inventory, source-type canary, staged extraction, current
  coverage, partial coverage, and complete terminal disposition.
- Detect added, modified, moved, deleted, and newly reachable occurrences;
  invalidate affected scope decisions and downstream evidence, then
  incrementally re-triage only changed items and declared dependents.
- Persist source content, metadata, analysis work, revisions, dependencies,
  discovery cursors, dispositions, recomputation state, and current projections
  under the external `MATTERS_HOME` root.
- Preserve immutable source versions and precise line/character, page,
  sheet/cell, message/thread, attachment, image-region, OCR, and metadata
  anchors.
- Run text and multimodal AI plus the single ResearchGuard research entrypoint
  through a structured, versioned, injectable `agent_operation` contract with
  visible pending-integration, unavailable, failed, stale, partial,
  scope-incompatible, quarantined, and current outcomes.
- Ship Matters-owned self-maintenance procedures as an app-local consumer Skill
  Pack; resolve a validated active view against compatible machine-installed
  skills without requiring or silently mutating global installation.
- Make admission, state, blocking, and outcomes evidence-licensed decisions
  with explicit source-only, candidate, uncertainty-preserved, not-applicable,
  no-delta, and blocked branches.
- Automatically validate and dispatch typed advisory findings to the unique
  original C4-C9/C12 owner without requiring user confirmation; preserve user
  correction as an optional later action.
- Complete original-owner recomputation before publishing a corrected
  projection.
- Provide one runnable local service path shared by HTTP, MCP, CLI, the
  browser development surface, and the primary bilingual Windows desktop UI.
- Derive changed modules, fields, code contracts, exhaustive finite cases, UI
  transitions, and test ownership from the revised models.
- Prevent credentials, secrets, browser/session stores, operating-system and
  application internals, caches, dependency/build outputs, VCS internals,
  executable launch, unsafe deserialization, uncontrolled archive expansion,
  and link/junction escape from entering the analysis path.
- Make packaging and local installation physically independent of private
  runtime and evaluation data.
- Freeze and verify source, model, test, package, installed projection, and
  local Git identities before accepting the v0.2 release candidate.
- Validate safety with synthetic cases, usefulness with per-type real private
  canaries, and coverage through a progressive first run over the authorized
  source universe without publishing any private source or derived user model.

**Non-Goals:**

- Reading credentials, secrets, system/application internals, caches, VCS
  internals, dependency/build outputs, or files outside declared content roots.
- Executing discovered files, loading unsafe serialized models, recursively
  expanding untrusted archives without limits, or following link/junction
  targets outside a declared root.
- Jira or Rovo login, Jira workflow replacement, bulk Jira import, or Jira as
  a v0.2 acceptance dependency.
- Sending Gmail, deleting or archiving messages, changing labels, or taking
  another external action solely because read access was authorized.
- Claiming `cloud_placeholder` content was read when it was not hydrated, or
  treating unsupported/audio/video/archive formats as equivalent to extracted
  text.
- Calendar and arbitrary third-party provider connectors beyond declared
  Gmail and local/cloud-backed user-content roots.
- Unsupported or unbounded truth, relationship, causal, completion, or
  priority inference without declared evidence, uncertainty, and owner rules.
- AI or local skills writing canonical Matter, lifecycle, outcome, identity,
  or evidence state.
- Silent fallback from unavailable AI/local skills to keyword heuristics while
  claiming equivalent analysis.
- Multiple compatibility success paths or UI/provider-owned canonical state.
- Publishing renamed, pseudonymized, hashed, embedded, summarized, screenshotted,
  or otherwise derived real private material as synthetic data.
- Public remote publication or an open-source license choice without a separate
  explicit user decision.

## Decisions

### 1. Extend existing owners instead of creating a second product

`M0_matters_end_to_end_authority` retains the end-to-end token and obligation
flow. C1-C12 retain unique ownership of authorization, source versions,
evidence, identity, temporal trace, admission, lifecycle, open loops, outcomes,
correction, advisory analysis, and projection. Local-source, persistence,
agent-operation, HTTP, MCP, CLI, and UI paths are adapters or orchestration
around those owners; they do not introduce new canonical state machines.

A separate “AI Matter model” or page-owned workflow was rejected because it
would duplicate C2-C12 state and make later correction irreconcilable.

### 2. Make the source universe explicit, staged, and finite

One source-universe grant records declared local user-content roots, the Gmail
read scope, configured cloud-backed content roots, allowed operations,
hard-exclusion policy, start revision, and revocation state. Pasted text and
single-file selection remain valid narrow grants. A root grant authorizes
bounded enumeration beneath that root but never authorizes link/junction escape
or reading system, credential, cache, dependency, build, VCS, or application
internal paths.

Discovery is paged and checkpointed. It first records metadata and a stable
occurrence identity, then assigns every item a current disposition. Before
broad extraction, a scope-organization gate freezes the candidate inventory
revision and classifies occurrences into `tracked`, `not_tracked`,
`hard_excluded`, `metadata_only`, `blocked`, `unavailable`, or another
declared state. A `not_tracked` decision retains a reason, policy revision,
decision owner, model/tool identity when AI participated, confidence,
privacy-safe feature references, and the conditions that make it stale.

Filesystem paging is not allowed to hide an unbounded in-memory tree walk. If
one root exceeds its frozen inventory budget, a private durable partition
manifest replaces the failed whole-root attempt. The parent partition records
only its direct files and safe immediate child-directory boundaries. Each child
is first inventoried as one bounded recursive subtree; only a child whose
bounded scan exhausts the frozen budget is shallow-partitioned again. This keeps
ordinary nested folders in one scope instead of creating a manifest node for
every directory. A parent boundary receives reversible
`not_tracked` with `covered_by_declared_child_scope`; it is coverage-routing
evidence, not proof that the child's contents were read. The aggregate root can
be complete only when the exact manifest revision, every required child
partition, and every child occurrence are current and terminal. This bounded
tree preserves the original root grant while avoiding a second authorization
owner or a whole-tree memory dependency.

Hard system, credential, browser/session, VCS, cache, dependency, build, and
unsafe-execution exclusions are deterministic policy decisions. Relevance,
duplicate-clutter, and junk triage are decided by AI under the frozen product
policy when the decision is reversible and auditable. Low-confidence or
potentially valuable items receive the best supported current disposition plus
confidence, alternatives, and uncertainty codes instead of waiting for user
approval. Scope-expanding requests fail closed. AI cannot enlarge
authorization, follow a blocked link, delete source content, or make an item
disappear without a disposition. Optional user overrides append a new decision
revision.

Filesystem partitioning applies the same current policy before it creates child
work. Known software-control, dependency, cache, build, test, and temporary
directories are terminal at their directory boundary, so their descendants do
not inflate coverage or semantic work. Every private partition manifest binds
the numeric tracking-policy revision; a revision mismatch is stale evidence
and cannot be resumed or counted as current.

The installed Windows default uses one non-virtualized directory directly
under the user profile. This avoids Windows Store LocalAppData virtualization
splitting Python-managed manifests and native SQLite state across different
physical paths. `MATTERS_HOME` remains the sole runtime authority, and every
database, manifest, receipt, managed skill, blob, and desktop profile must
resolve beneath that one physical root.

Extraction runs by source-type canary and staged expansion only for `tracked`
items. Complete coverage requires terminal page/cursor evidence and one of
`ingested`, `not_tracked`,
`hard_excluded`, policy-authorized `metadata_only`, `unsupported`,
`excluded_sensitive`, `inaccessible`, `changed_during_read`,
`cloud_placeholder`, `quarantined`, `revoked`, `deleted`, or another explicitly
declared terminal disposition for every enumerated item. Every non-ingested
terminal remains visible and cannot be counted as ingested or analyzed.

A filesystem content canary reuses the current partition manifest and selects
the smallest tracked complete partition that can fill the remaining sample
first; if none can, it chooses the largest useful partial partition. It neither
chooses an arbitrary giant subtree nor crosses many single-file scopes merely
to obtain a few content samples. Canary completion remains separate from
whole-root semantic coverage.

Within that canary, per-item source, evidence, analysis, and projection stage
updates are persisted without recomputing the global coverage summary. The
bounded batch performs one summary refresh only after all selected items reach
their current terminal join.

Each committed directory or mailbox inventory has a snapshot revision derived
from provider occurrence identity and policy-approved metadata, not a public
path or content hash. Later scans produce an explicit `added`, `modified`,
`moved`, `deleted`, `unchanged`, or `newly_reachable` change set. A changed
occurrence stales its prior triage/extraction/analysis outputs and only that
occurrence plus declared dependents is reprocessed; an unchanged occurrence
produces no duplicate work. Tracking-policy changes stale the affected
classification set even when source bytes did not change.

For Gmail, the grant freezes account identity, query/policy revision, mailbox
categories, allowed operations, pagination cursor, and attachment policy.
Inbox, Sent, and archived non-Spam/non-Trash mail are progressively read and
dispositioned. Spam and Trash are `hard_excluded` or `metadata_only`. Read scope does
not permit sending, deletion, archiving, or label changes. The prior
shortlist-only pilot design was rejected because it cannot satisfy the
authorized first-run coverage claim. Spam/Trash are hard policy exclusions;
other categories such as Promotions are not assumed to be junk and pass
through the frozen relevance policy. Inside a bounded canary, current tracked
messages whose authorized connector page actually supplies body content are
selected before tracked metadata-only messages; the exact content budget still
applies. Opaque message identifiers are provenance identities, not a content
priority signal.

### 2a. Keep AI scope triage distinct from authorization and product truth

C1 owns the grant, frozen candidate scope, hard exclusions, tracking policy,
and current tracking disposition. C2 owns occurrence identity, inventory
snapshots, change sets, and freshness invalidation inputs. C11 executes the
versioned AI triage work package and records the bounded result. M0 routes the
result back to C1; C11 never changes grants or writes a C1 disposition
directly. C12 shows `tracked`/`not_tracked`/`hard_excluded`/`metadata_only`/
`blocked` counts, reasons on demand, stale classifications, changed-item
queues, and optional user override controls.

This partition lets AI perform the user's requested first-pass organization
without turning a model guess into silent authorization expansion or
irreversible data loss. Treating every authorized item as automatically useful
was rejected because it wastes analysis and hides clutter. Treating AI output
as an unreviewable delete/exclusion command was rejected because freshness,
false exclusion, and correction could not be repaired.

### 3. Use one durable local store under MATTERS_HOME

SQLite stores grants, candidate-scope and tracking-policy revisions, discovery
cursors, inventory snapshots, change sets, triage decisions, per-item
dispositions, metadata, source/version identities, evidence anchors, advisory
artifacts, user decisions, revisions, dependency edges, recomputation work,
and current projection references. BlobStore stores immutable source bodies
and private derived artifacts under `MATTERS_HOME`. The repository contains
schemas and fully synthetic tests only; it never stores live rows, blobs,
excerpts, embeddings, screenshots, private paths, or private content hashes.

Every durable write is append-oriented or versioned. Service startup requires a
configured external `MATTERS_HOME`; tests use explicitly created temporary
external roots. Restart recovery rebuilds current views from durable records
and resumes only work whose authorization and input identities remain current.

An in-repository database, implicit temp fallback, or in-memory-only success
path was rejected because it breaks privacy and restart semantics.

### 4. Define a real text and multimodal agent-operation boundary

The core creates an `AnalysisWorkPackage` containing authorization id,
SourceVersion ids, minimized content or private blob handles, exact
EvidenceAnchors, requested operation ids, modality, resource budget, a
deterministic private input fingerprint, prompt-contract/schema/policy/owner/
locale-registry revisions, required skill/model/runner identities, allowed
evidence/asset/tool ids, and `autonomy_mode="auto_apply"`. An injected
`AgentOperationRunner` returns versioned `AgentOperationResult` rows for text
AI, multimodal image/OCR analysis, and an abstract `ResearchOperation`.
ResearchGuard is the intended single provider for that research operation
after its separate SourceGuard, TraceGuard, and LogicGuard merger is frozen and
validated. Each result records runner identity, operation id, tool/skill
version, status, input fingerprint, output, gaps, and evidence references.

Source content is untrusted data. Work packages isolate prompt-like source text
from controlling instructions, prohibit undeclared tools and adjacent-source
reads, minimize content sent to any external model, and return quarantined or
scope-incompatible status when the boundary is challenged.

The initial production boundary is the app-local Matters consumer Skill Pack
and MCP/CLI facade. Unit and synthetic tests inject a deterministic fake
runner. While ResearchGuard is not yet frozen and validated, the real research
operation returns `researchguard_pending_integration`; this does not block
  non-research source inventory, autonomous non-research processing, optional
  inspection/correction, or synthetic conformance work,
but it blocks claims that real research analysis completed. When another
runner is absent, the product returns `analysis_unavailable`; it never pretends
keyword heuristics or three legacy parallel bindings are equivalent evidence.
Canonical owners may consume only current, scope-compatible advisory
candidates.

Direct model/API calls from canonical owner modules were rejected because they
would hide tool identity, authorization, and freshness.

### 4a. Bundle self-maintenance skills and resolve one validated active view

Matters ships an immutable, app-local consumer Skill Pack alongside the
matching runtime code. The initial pack contains eleven required consumer
skills:

- `matters-source-governance`;
- `matters-inventory-reconciliation`;
- `matters-freshness-maintenance`;
- `matters-model-depth-maintenance`;
- `matters-human-correction`, which handles optional post-publication
  inspection and correction without becoming a normal modeling gate;
- `matters-model-miss-review`;
- `matters-skill-runtime`;
- `matters-research-orchestration`, which invokes only the abstract
  `ResearchOperation` and, after the external integration gate passes, the
  single ResearchGuard provider; and
- `matters-semantic-understanding`, which consumes only bounded
  EvidenceAnchor-whitelisted work packages and returns same-revision
  `en`/`zh-CN` advisory candidates;
- `matters-autonomous-maintenance`, which advances the durable coverage ledger
  by dispatching the next missing or stale stage without owning another
  component's canonical fields; and
- `matters-card-visual-curation`, which selects one representative visual from
  an allowlisted current candidate set and returns bilingual display text
  without creating or fetching evidence.

Each consumer skill calls the same MatterService/CLI/API operations as the UI.
It cannot read undeclared sources, enlarge grants, or write canonical owner
state. The pack contains consumer `SKILL.md`, scripts,
references, assets, matching runtime/engine code, and a manifest with skill id,
version, schema compatibility, Matters compatibility, origin, content hash,
required/optional status, install policy, capabilities, permissions, disclosure
policy, dependencies, and native validator. It excludes author-side
`.skillguard` contracts, receipts, router state, and maintenance control data.

Document, Gmail, and image extraction finishes before semantic understanding.
For each current SourceVersion, M0 chunks current anchors into bounded private
work packages. The semantic-understanding operation may emit typed Matter,
person, event, deadline, open-loop, conflict, completion-gap, or summary
candidates, but every finding must cite only the package whitelist and persist
equivalent `en` and `zh-CN` statements on the exact source revision. Every
input receives exactly one typed disposition. Missing locales, stale
revisions, adjacent-source requests, or out-of-package evidence fail the
result. Passed results remain advisory but are automatically dispatched by a
durable deterministic service to the declared original C4-C9/C12 owner. The
owner validates and writes its own state; no user confirmation or AI-owned
canonical path exists.

The resolver has exactly three layers: the immutable bundled pack, the
machine-installed inventory, and the resolved active view. A
Matters-managed installed projection is a managed ownership subtype within the
machine-installed layer; it is not a fourth layer. Each candidate uses a PEP
440 version, a declared compatible interval for Matters and the skill schema,
an explicit prerelease-acceptance flag, origin, content hash, runtime identity,
dependency identities, native-validator identity, and ResearchGuard identity
when applicable.

1. If no machine-installed copy exists, use the bundled copy internally and do
   not install it globally.
2. If local and bundled version plus content hash match, select that exact
   identity deterministically.
3. Compare otherwise eligible versions using PEP 440 after compatibility and
   native validation. A prerelease is eligible only when its manifest
   explicitly opts into prerelease use for the current Matters interval.
4. If the eligible machine-installed copy is newer, select it as a
   non-mutating overlay.
5. If the bundled copy is newer, use it internally. A machine-installed update
   is allowed only for the managed subtype previously installed and marked as
   Matters-managed; stage, validate, activate transactionally, and restore the
   prior projection on failure. Externally managed local skills stay unchanged
   and are reported as update-available.
6. A same-version/different-hash collision, an invalid manifest, or zero
   validated compatible candidates blocks that required skill instead of
   guessing, downgrading silently, or using lexical version order.

Selection means the highest current validated compatible candidate, not merely
the numerically highest version. A change to any selected manifest, runtime,
dependency, validator, or ResearchGuard identity invalidates the active view
and requires resolution plus native validation again before use. Package bundle
bytes are never rewritten at runtime; a later Matters release supplies a newer
bundle.

An auxiliary FlowGuard owner
`S0_matters_skill_runtime` models this infrastructure as S1 inventory, S2
compatibility, S3 active resolution, S4 Matters-managed synchronization, and
S5 validation/rollback. It is not C13 and owns no Matter, evidence, lifecycle,
or projection field. Product-semantic freshness and depth are maintained
through C1/C2/C10/C11/C12; software behavior or FlowGuard model changes enter
the explicit Model Miss/development pipeline and are never live self-edits.

Semantic depth is a revision-bound product assessment with states
`not_assessed`, `partial`, `sufficient`, `blocked`, and `stale`. `sufficient`
requires every policy-required tracked occurrence and requested analysis class
to have current anchored evidence or an explicit non-applicable disposition,
with no unresolved blocking gap. `partial` means some useful current analysis
exists but the sufficient criteria are not met. `blocked` means a required
  source, extractor, ResearchGuard operation, owner dispatch, or projection
  cannot terminate.
Changing source/inventory, tracking policy, anchor, model/provider, operation
  schema, dependency, or user-correction identity makes the affected
  assessment `stale`.
C12 shows the state, unmet criteria, freshness revision, and next action; it
never projects partial, blocked, or stale depth as complete modeling.

The final frozen bundle contains ResearchGuard only after its external consumer
pack, engine, manifest, compatibility declaration, and native validation are
current. Separate SourceGuard, TraceGuard, and LogicGuard bindings may be read
only as explicitly stale/source-only migration evidence; they are not parallel
runtime fallbacks. A missing current ResearchGuard permits inventory,
autonomous non-research processing, optional inspection/correction, synthetic
work, and a blocked local candidate, but it blocks both the
ResearchGuard-complete analysis claim and the final v0.2 complete-release claim.

### 5. Complete recomputation before projection publication

Corrections, source updates, revocation, and deletion append a new revision and
produce a durable invalidation plan. C10 dispatches recomputation to the
registered C3-C9 owners in dependency order. Each owner produces a current
completion or blocked disposition. C12 publishes a new projection only after
every required disposition is terminal and the lifecycle/outcome owners have
recomputed; otherwise it publishes a clearly non-canonical pending status
outside board totals.

The former behavior of creating request objects and immediately publishing a
fresh-looking projection was rejected as an evidence-overclaiming model miss.

### 6. Keep provider envelopes generic and Jira dormant

All local files, documents, images, mail messages, attachments, cloud
placeholders, and pasted inputs become generic source occurrences and
SourceVersions or explicit non-ingested dispositions. Jira identity, when a
future change explicitly activates it, remains an
`ExternalReference(provider="jira")`. The existing Jira adapter is disabled by
default, has no core-required fields, and cannot write Matter or lifecycle
state. Atlassian Rovo is not used.

Deleting the generic adapter was rejected because it is already isolated and
does not burden the v0.2 path. Activating it in v0.2 was rejected because it
would reintroduce an unnecessary external dependency.

### 7. Expose one service path through CLI, HTTP, MCP, and UI

`MatterService` remains the application facade. Durable catalog/repository
adapters, bounded discovery/extraction adapters, and an injected agent runner
are its dependencies. CLI, HTTP, MCP, and UI call the same service methods for
health/capability, source-universe grants, inventory/disposition progress,
pasted and selected-file input, staged discovery/extraction, analysis, listing,
detail, optional correction, revocation, deletion, recomputation status,
representative-visual intents, and bilingual projection. The plugin manifest must
enumerate only implemented methods.

The primary UI is a Windows desktop application that embeds the packaged web
surface and starts the local service plus bounded durable worker. The browser
surface remains runnable for development and accessibility verification but
does not become a second business path. Every enabled control has a
`control -> event -> owner -> function -> UI update -> evidence` chain,
including cancel, failure, retry, keyboard, and recovery paths.

Separate “demo” and “real” business routes were rejected. Synthetic behavior
is obtained by injecting synthetic providers and runners into the same facade.

### 8. Preserve one semantic revision across locale-keyed views

Canonical machine fields, evidence references, and semantic revision identity
remain language-neutral. Every user-visible AI-authored label, summary,
rationale, explanation, gap, blocker, forecast, uncertainty explanation,
correction prompt, and next action
is persisted as a `LocalizedText` value keyed by a registered BCP 47 locale.
Version 0.2 requires non-empty `en` and `zh-CN` values bound to the same
semantic revision and evidence references. Static UI chrome is held in the
same locale registry, but it is not evidence and cannot change canonical
meaning.

`en` is the first-launch default. A user can select `zh-CN`; the preference is
stored locally and changes only the display projection, not the semantic
revision, current task, focus target, or selected evidence. CLI, HTTP, MCP, and
the browser UI consume the same locale registry and expose the available and
default locales instead of maintaining separate hard-coded language paths.

Later languages are added by registering a locale and supplying a complete,
validated localized projection for that locale. Registration does not create
a new fact authority and does not mutate existing semantic meaning. A missing,
empty, stale, conflicting, or unregistered required locale is a visible
localization gap and blocks publication for that locale; the normal runtime
does not silently substitute another language.

The UI can initiate authorization, analysis, optional correction, revocation,
deletion, and representative-visual intents but cannot write canonical state.
Evidence detail is user-on-demand and machine-local identifiers remain
internal. A localization conflict, stale analysis result, or pending
recomputation is visible and blocks canonical board publication.

### 8a. Reuse the Databank shell as the object-browser design authority

The frozen Databank UI handoff owns Matters v0.2 color, typography, spacing,
left navigation, card, detail-dialog, and interaction geometry. Matters changes
only the brand/logo asset and maps modeled Matter content into those
components. A Figma component library and desktop frames mirror the same source
authority; Figma is a design and comparison surface, not a separate product
implementation.

The right content region renders Matter cards; the left navigation filters by
state, time, people, topic/type, and source type. The current primary detail
sections are Overview, Timeline, People, Actions and open loops, Related
Matters, Images, Files and messages, and Evidence and change history. Internal
model ids, receipts, paths, connector ids, and debug metadata remain hidden.

`ui.card_density` is an independent persisted preference with values
`standard` and `compact`. It is not the responsive viewport mode and never
changes canonical state, sorting, filtering, selected Matter, locale, semantic
revision, visual decision, focus ownership, or freshness. Standard preserves
the handoff's three-column desktop card geometry and secondary metrics;
compact preserves the same hero/title/state while using smaller geometry and
hiding low-priority metrics. The current one-card vertical-stretch behavior is
a source defect to repair without redesigning the shell.

### 8b. Give every Matter one current representative-visual disposition

C2 owns original and derived visual versions, C3 owns the exact page/region
anchor, C6 owns the Matter-source relation, C11 owns the advisory multimodal
selection, and C12 owns the current display decision. Eligible candidates are
current authorized related photos, existing images, mail/document attachments,
embedded images, and offline deterministic document/page/slide/sheet previews.
Renderers disable macros, active content, external links, and network access.

Safety and permission filtering occurs before AI selection. When the allowlist
is non-empty, AI must choose one candidate. A deterministic compiler validates
the result, preserves a still-valid current or user-pinned choice, and resolves
AI failure or ties through declared relation, asset-kind, quality, and stable
hash ordering. With zero eligible candidates it publishes a neutral
placeholder. It never searches the Internet or generates decorative imagery
that could be mistaken for evidence. Source, relationship, renderer, policy,
model, prompt, or user-correction changes stale the decision and schedule
bounded recomputation.

### 8c. Make completion machine-auditable through ObjectCoverageLedger

One private durable ledger row exists for every registered occurrence and
admitted Matter. It records authorization, discovery, tracking disposition,
source version, extraction, analysis, original-owner terminal, semantic depth,
required localization, representative-visual disposition, UI reachability,
freshness, retries, and terminal reason. The worker repeatedly chooses a
missing or stale stage and delegates it to the existing owner until the row is
UI-ready or genuinely blocked. The ledger coordinates completion but owns no
canonical source, Matter, lifecycle, outcome, or projection field.

### 9. Separate three security and evidence domains

The Git repository contains public source, models, schemas, and fully synthetic
fixtures. `MATTERS_HOME` stores live private runtime data.
`MATTERS_EVAL_VAULT` stores explicitly selected private evaluation evidence.
No automatic copy crosses domains.

All first-run raw content and derived output stay under `MATTERS_HOME` and,
when explicitly frozen for evaluation, under `MATTERS_EVAL_VAULT`. This
includes local paths, file metadata, message ids, addresses, subjects, bodies,
links, booking/payment references, codes, excerpts, private content hashes,
embeddings, OCR text, image labels/regions, screenshots, user models, logs, and
receipts that can reconstruct private material. Public receipts contain only
fully synthetic details or non-reconstructable aggregate outcomes, bounded
failure classes, and opaque private evidence handles.

Public evidence uses repository-relative paths, portable interpreter/tool ids,
and deterministic source fingerprints that exclude ignored files, caches,
bytecode, logs, and private roots. The public checker decodes structured files,
detects escaped local paths and identity fragments, verifies required-public
files are not ignored, and retains G11 package/history limits.

### 10. Enforce a revised gated delivery sequence

The active sequence is:

1. rebase OpenSpec, G0 privacy/inventory evidence, and the source-universe
   policy; mark earlier bounded-input evidence stale;
2. rebase G1 product, agent-operation, and development-process commitments;

## Risks / Trade-offs

- **External AI/local skills may be unavailable** → Return a visible
  `analysis_unavailable` or failed operation; retry within budget and use only
  an explicitly declared deterministic selector for bounded decisions such as
  visual ties. Preserve source access and honest partial/blocked status without
  claiming equivalent AI completion or asking the user to approve each item.
- **ResearchGuard source exists but its consumer migration and installation are
  not yet closed** → Keep the real provider behind
  `researchguard_pending_integration`, develop and test the abstract contract
  deterministically, and integrate only one frozen current suite identity. A
  portable currentness receipt records `checked_at`, source commit,
  distribution, command, top-level skill, member projections, manifest,
  residual state, and terminal disposition without machine-local paths.
- **Bundled and machine-installed skills can disagree** → Resolve one validated
  compatible active identity by manifest and content hash, mutate only a
  Matters-managed projection transactionally, and block collisions instead of
  guessing.
- **Natural-language output is nondeterministic** → Bind every result to exact
  inputs, runner/tool identity, schema validation, anchored evidence, and
  automatic canonical-owner validation; keep synthetic runner cases
  deterministic and preserve uncertainty instead of requiring confirmation.
- **SQLite and blob state can diverge** → Use one transaction boundary for
  metadata plus staged blob activation and verify references at startup.
- **Corrections can cause wide recomputation** → Persist exact dependency edges,
  dispatch only affected owners, and keep the projection pending until required
  terminal dispositions exist.
- **Model and evidence volume can grow quickly** → Keep child state finite,
  use durable page cursors, bounded concurrency, quotas, per-type canaries,
  backpressure, and one owner per write; reserve one broad verification for the
  frozen candidate.
- **Recursive discovery can escape or loop** → Normalize and resolve every
  path, block symlink/junction escape, track stable occurrence identities, and
  make cycle/permission/change-during-read outcomes explicit.
- **Cloud folders can contain unhydrated placeholders** → Distinguish metadata
  presence from readable content and never claim content coverage until the
  provider supplies a stable readable version.
- **Archives and serialized models can be hostile** → Inventory metadata first,
  enforce expansion/decompression/resource limits, quarantine unsafe formats,
  and never execute or deserialize discovered artifacts on the main path.
- **Documents and mail can contain prompt injection** → Treat source text as
  untrusted payload, isolate it from runner instructions, deny undeclared tools,
  and validate every returned anchor and scope.
- **OCR, EXIF, image labels, and face similarity are uncertain** → Keep pixel,
  OCR, EXIF, inferred label, and identity evidence distinct; never treat EXIF
  time/location or visual similarity as event/identity proof on its own.
- **UI can expose private or internal identifiers** → Classify every displayed
  item as user-visible, user-on-demand, or internal and validate DOM/runtime
  evidence.
- **Real local and Gmail sources contain credentials, signed links, booking
  references, personal identifiers, and private derived meaning** → Apply hard
  exclusions and content minimization, keep raw and derived material outside
  Git, and publish only synthetic evidence or non-reconstructable aggregates.
- **Public scanners can miss encoded machine identity** → Decode JSON/YAML and
  scan normalized plus raw representations; test escaped Windows/POSIX paths.
- **The current FlowGuard project audit has an external suite-inventory
  blocker** → Keep it visible, run target-owned model checks directly, and
  never convert the disposition into broad project-audit green evidence.
- **No public license has been selected** → Produce a local source release
  candidate only; public remote release remains out of scope until explicit
  legal/product direction.
